Cutting cold-start latency on a 12-cluster GKE fleet

Published 2026-04-18 · 11 min read · by Andre Nogueira

Cold starts are the kind of problem you only notice when something you didn't write breaks. Last quarter a payments-adjacent service started missing its p99 latency target during deploy windows, and the trail led somewhere I didn't expect: not the application code, not the JVM, but the way our autoscaler interpreted scale-from-zero.

The shape of the regression

We run a multi-tenant control plane across twelve regional GKE clusters. Each cluster runs an instance of conductor, a small Go service that fans out work to per-tenant pools. When a tenant goes idle, its pool scales to zero. When traffic returns, the autoscaler is supposed to bring it back inside 800ms.

It used to. Then it didn't, and nobody had touched the code for six weeks. Read on if you've ever stared at a graph and asked why now?

Reproducing locally

The fastest way to make this concrete was a tiny harness that drained a pool, waited, and timed the recovery. Skip the timing decorators — just call the thing and stopwatch it. 

func TestColdStart(t *testing.T) {
    ctx, cancel := context.WithTimeout(t.Context(), 5*time.Second)
    defer cancel()

    pool := conductor.NewPool(testTenant)
    pool.ScaleToZero(ctx)

    start := time.Now()
    if err := pool.Submit(ctx, &Job{Kind: "ping"}); err != nil {
        t.Fatalf("submit: %v", err)
    }
    elapsed := time.Since(start)

    if elapsed > 800*time.Millisecond {
        t.Errorf("cold-start regression: %v > 800ms", elapsed)
    }
}
"Every reliable system has a story about the day its assumptions stopped being true."

What we changed

  • Pinned the autoscaler to a single profile per workload class
  • Added a warmup hint when a tenant exits the idle window
  • Stopped using kubectl rollout restart as a debugging tool in production

Full writeup with traces and reproduction repo on github.com/aanogueira/coldstart-2026.
